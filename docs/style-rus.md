## Стиль русской документации

Этот файл описывает, как писать русские README и доки в этом репозитории, чтобы текст был живым и понятным разработчикам, а не выглядел как машинный перевод.

### 1. «Зачем» vs «чем / в чём»

- Не используем заголовки вида «Зачем этот проект важен», «Зачем это важно».
- Вместо этого:
  - «Чем этот проект важен»,
  - «В чём важность/ценность проекта»,
  - «Что делает этот проект полезным».

### 2. Меньше абстракций, больше нормального языка

- Избегаем голых абстрактных существительных:
  - плохо: «высокая скорость тестового цикла», «устойчивость локаторов», «гибкость управления наборами»;
  - лучше: «быстрый прогон тестов», «локаторы не ломаются от мелких правок», «можно запускать разные группы тестов».
- Стараемся говорить прилагательными и глаголами, а не «‑ость»:
  - не «повышает устойчивость», а «локаторы реже отваливаются»;
  - не «обеспечивает наблюдаемость», а «пишет результаты в SQLite, файл прикрепляется к артефактам CI».

### 3. Лексика: говорить как живой разработчик

- Нормальная тех‑лексика:
  - тесты, прогон, фреймворк, локаторы, логин, сессия;
  - пайплайн, CI, джоба;
  - смоук, full run, e2e, API‑тесты.
- Избегаем канцелярита и маркетинга:
  - не «повышает значимость проекта»;
  - не «гибкое управление наборами»;
  - не «устойчивость локаторов»;
  - вместо этого описываем, что именно удобно: «тесты размечены маркерами, можно запускать только smoke / только notes / только e2e».

### 4. Конкретика вместо воздуха

- Любое «улучшает / повышает / обеспечивает / упрощает» должно сопровождаться «как именно»:
  - «убирает повторный логин, за счёт этого прогон быстрее»;
  - «локаторы строятся на `data-testid` и `role`, поэтому правки верстки реже ломают тесты».

### 5. Простые формулировки

- Если можно сказать проще — говорим проще:
  - `быстрый прогон` вместо `обладает высокой скоростью тестового цикла`;
  - `удобно запускать нужные группы тестов` вместо `предоставляет гибкий механизм управления наборами`.

### 6. Заголовки

- Заголовки должны быть понятны без пояснений и звучать так, как сказал бы разработчик:
  - «Чем этот проект важен» / «Что делает проект полезным»;
  - «Как устроены локаторы» / «Почему локаторы не ломаются»;
  - «Как запускать разные группы тестов».
- Не придумываем «красивые» формулировки вроде:
  - «Зачем этот проект важен»,
  - «Гибкое управление наборами»,
  - «Устойчивость локаторов».

### 7. Мини‑таблица «было → стало»

Ориентиры по перефразированию:

| Было (не так)                           | Стало (так)                                          |
| --------------------------------------- | ---------------------------------------------------- |
| Зачем этот проект важен                 | Чем этот проект важен / В чём важность проекта       |
| Высокая скорость тестового цикла        | Быстрый тестовый цикл / Тесты бегут быстро           |
| Устойчивость локаторов                  | Надёжные локаторы / Локаторы не ломаются от правок   |
| Гибкое управление наборами              | Можно запускать разные группы тестов через маркеры   |
| Обеспечивает наблюдаемость              | Пишет результаты в SQLite, база прикрепляется в CI   |
| Обеспечивает высокую производительность | Убирает повторный логин, за счёт этого прогон быстрее |
